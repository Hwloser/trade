# Trade 项目进化计划

**版本：** 2.0
**日期：** 2026-03-01
**来源：** 头脑风暴会话 2026-03-01
**目标：** 从 A 股量化工具 → 个人多资产智能决策支持系统

---

## 一、新愿景

> **"每天 9:20，用5分钟读完系统生成的作战简报，清楚知道今天关注什么、什么条件出手、什么情况不动。剩下的时间，让系统替你守夜。"**

系统不是价格预测机，而是一个**有对抗性的决策伙伴**：
- 它帮你找时机，不帮你猜价格
- 它挑战你的判断，不迎合你的偏见
- 它告诉你为什么，不只告诉你买还是卖
- 它主动说"不做"，而不是总在找信号

## 二、架构总体设计

### 2.1 信号流水线

```
数据采集层（Python / akshare + RSS + 公告爬取）
        ↓ 每日 T+1 批量 + 盘中定时
规则引擎层（C++ 快路径，每5分钟）
        ↓ 触发阈值时唤醒
分析综合层（Python + LLM，生成判决书）
        ↓
晨报生成器（每日 9:20 前，Markdown 输出）
        ↓
用户交互层（本地文件 / Telegram Bot / CLI）
```

### 2.2 三大信号域

| 信号域 | 核心问题 | 主要数据源 |
|--------|----------|------------|
| **宏观联动域** | 地缘事件/行业冲击 → 哪些资产受益？ | RSS、官方发言、政策文件、美股夜盘 |
| **博弈行为域** | 机构在哪、散户在哪、窗口在哪？ | 大单资金流、换手率、融资余额、舆情 |
| **个人决策域** | 现在是你应该出手的时刻吗？ | 用户持仓、历史决策日志、窗口质量分 |

### 2.3 输出原则

每个信号必须包含：
```
概率：XX%（置信区间 XX%-XX%）
根因：[主证据链，最多3条]
反驳：[反向证据，至少1条]
窗口质量：XX/100
建议行动：[具体条件句，"若…则…"]
```

### Phase 7：晨报引擎 + 信号框架重构

**目标：** 建立每日晨报工作流，替代现有分散的信号输出

#### 7.1 晨报生成器（Python）

**新文件：** `python/trade_py/morning_brief.py`

- 每日 9:10 执行（cron 或手动触发）
- 输出结构：
  ```
  ## 今日作战简报 YYYY-MM-DD
  ### 宏观环境 [评分 X/10]
  ### 重点关注标的（最多3只）
    - 标的 + 窗口质量分 + 出手条件
  ### 今日建议：不做（附理由）/ 关注以下条件
  ### 昨日预测回溯（命中/未命中）
  ```
- 输出到 `data/briefs/YYYY-MM-DD.md`

#### 7.2 窗口质量评分引擎（C++）

**新文件：** `include/trade/decision/window_scorer.h` / `src/decision/window_scorer.cpp`

- 输入：技术信号 + 资金流信号 + 宏观环境分
- 输出：`WindowScore { score: int, level: enum{Watch,Alert,Act,DoNothing}, reasons: vector<string> }`
- `DoNothing` 是最重要的输出——当无高质量窗口时明确返回此状态

#### 7.3 "不做"信号优先级设计

- 默认状态为 `DoNothing`
- 只有当三层信号（宏观层≥6分、标的层≥70分、个人层≥60分）**同时满足**才升级
- 晨报默认第一行：**"今日建议：观望"**，除非明确升级

#### 7.4 快慢双引擎 cron 调度

**新文件：** `python/scripts/scheduler.py`

| 时段 | 任务 | 频率 |
|------|------|------|
| 09:10 | 晨报生成 | 每交易日一次 |
| 09:25-11:30 | 盘中规则引擎扫描 | 每5分钟 |
| 13:00-15:00 | 盘中规则引擎扫描 | 每5分钟 |
| 15:10 | 收盘深度分析 + 次日预判 | 每交易日一次 |
| 21:00 | 夜盘宏观事件扫描（美股/大宗/政策） | 每日一次 |

### Phase 8：博弈论信号层

**目标：** 构建"谁在买卖"的信号体系，替代纯价格预测

#### 8.1 散户情绪顶点探测器

**更新：** `python/trade_py/fund_flow_fetcher.py`

新增指标：
- `retail_heat_score`：融资余额增速 + 讨论量增速 + 涨停追买率的合成分
- `distribution_zone_flag`：三指标同时创近N日新高时标记为"分发区间"

#### 8.2 机构建仓时序模式识别（C++）

**新文件：** `include/trade/features/smart_money_signal.h`

检测连续N日的建仓模式：
```cpp
struct SmartMoneySignal {
    bool large_order_turning_positive;  // 大单净流入由负转正
    bool turnover_rate_dropping;        // 换手率骤降（浮筹出清）
    bool price_consolidating;           // 价格在区间内震荡（发现完成）
    int  confluence_days;               // 三者共振持续天数
    float confidence;                   // 置信度
};
```

#### 8.3 洗盘 vs 真建仓区分

**更新：** `src/features/smart_money_signal.cpp`

- 真建仓特征：价格波幅 < 2%，成交量温和放大，买入价格分布均匀（熵值低）
- 洗盘特征：价格日内波幅 > 4%，大单买卖交替出现（自成交率高）

#### 8.4 信号纯度评分

**更新：** `include/trade/features/feature_extractor.h`

新增 Group G（纯度层）：
- `signal_purity`：信号来源数量的倒数（来源越少越纯）
- `crowd_awareness`：该信号在社交媒体的传播量（传播越广纯度越低，负向权重）
- `N_FEATURES` 从 55 → 57

#### 8.5 等待成本计算器

**新文件：** `python/trade_py/patience_tracker.py`

- 记录每只标的进入"等待建仓"状态的时间
- 基于历史相似形态估算平均等待期
- 每日晨报展示等待进度百分比和机会成本

### Phase 9：宏观跨资产联动层

**目标：** 实现地缘事件 / 行业冲击 → 跨资产信号的自动传导

#### 9.1 事件知识图谱升级

**更新：** `python/trade_py/sector_graph.py`（现有 SectorGraph 扩展）

扩展为三层图：
```
Layer 1: 事件节点（战争/制裁/科技突破/政策）
Layer 2: 资产节点（黄金/BTC/原油/板块/个股）
Layer 3: 边权重（历史命中率 + 置信区间）
```

- 每条边附置信度：由历史事件实际传播结果动态校准
- 传播衰减：每跳衰减 40%，最多 3 跳，低于阈值自动截止

#### 9.2 VIP 发言监控器

**新文件：** `python/trade_py/vip_monitor.py`

- 数据源：主要国家领导人 Twitter/X、官方声明 RSS
- NLP：提取"语言温度"（情绪强度 + 关键词激烈程度）
- 输出：`vip_tension_score`（0-100），超过阈值触发宏观预警

#### 9.3 声明预解析器

**新文件：** `python/trade_py/statement_parser.py`

- 提前抓取重要会议议程、央行发言日程
- 生成"预期内容向量"（市场共识预期）
- 发言后对比实际内容，计算 delta 作为真实信号
- 逻辑：预期 vs 实际的偏差量，而非发言内容本身

#### 9.4 BTC/黄金联动信号

**新文件：** `python/trade_py/cross_asset_signal.py`

- 输入：`vip_tension_score` + 外汇波动率（USDCNH）+ 原油异动
- 输出：黄金/BTC 的预期联动强度（0-100）+ 建议配置比例
- 进入晨报的"宏观环境评分"板块

#### 9.5 监管文件语气探测器

**新文件：** `python/trade_py/regulatory_tone_monitor.py`

- 数据源：证监会、银保监公告全文（akshare 或直接爬取）
- NLP：追踪措辞从"引导"→"规范"→"严格"的渐进变化
- 输出：`regulatory_cycle_stage`（宽松/中性/收紧）和变化趋势

### Phase 10：对抗决策引擎

**目标：** 系统不只给出结论，还挑战用户的判断，并输出可解释推理

#### 10.1 判决书格式输出（C++）

**新文件：** `include/trade/decision/verdict.h`

```cpp
struct Verdict {
    float probability;           // 0.0-1.0
    float confidence_low;        // 置信区间下界
    float confidence_high;       // 置信区间上界
    vector<string> pro_evidence; // 支持证据（最多3条）
    vector<string> con_evidence; // 反驳证据（至少1条）
    WindowScore window;          // 窗口质量
    string action_condition;     // "若X则Y"形式的行动条件
    string final_ruling;         // 最终裁定理由（一句话）
};
```

#### 10.2 魔鬼代理人

**新文件：** `python/trade_py/devil_advocate.py`

- 每次生成买入判决书，自动生成"反方陈述"
- 反方陈述包含：
  1. 最可能犯错的3个原因
  2. 历史上相似形态失败的3个案例
  3. 当前信号的反向证据
- 在晨报中与正方并列展示

#### 10.3 用户偏差档案

**更新：** `python/trade_py/decision_journal.py`（现有文件扩展）

新增分析维度：
- 按板块分析历史胜率偏差（哪个板块你系统性过度乐观？）
- 按市场环境分析错误模式（你在震荡市还是趋势市犯更多错？）
- 按持仓时间分析收益分布（你拿不住赚钱的股，拿得住亏钱的股？）

每次晨报底部附：**"你的历史偏差提醒"**（一句话）

#### 10.4 多人格投资委员会

**新文件：** `python/trade_py/multi_persona.py`

5个独立投资人格，每个独立评分：
| 人格 | 决策逻辑 |
|------|---------|
| 激进趋势追踪者 | 动量优先，顺势而为 |
| 极端逆向者 | 人多的地方不去，人少的地方找机会 |
| 纯基本面派 | 只看 PE/PB/ROE/现金流 |
| 纯技术派 | 只看 K线形态+量价关系 |
| 宏观宿命论者 | 政策和宏观决定一切，个股只是载体 |

**高置信信号定义：** 至少2个原本分歧的人格罕见达成共识

#### 10.5 预测回溯自动归因

**更新：** `python/trade_py/decision_journal.py`

- T+5/T+20/T+60 自动触发回溯对比
- 预测错误时，反向追溯：是哪条证据权重过高？是哪个人格的判断最准？
- 自动调整各人格的历史权重

## 四、技术边界原则（不变）

| 原则 | 说明 |
|------|------|
| **决策时 → C++** | `WindowScorer`、`SmartMoneySignal`、`Verdict`、`FeatureExtractor` 全部 C++ |
| **分析时 → Python** | LLM 调用、晨报生成、另类数据采集、图谱更新全部 Python |
| **LLM 按需唤醒** | 只在规则引擎触发阈值时调用 LLM，不做实时流式分析 |
| **固定时间交付** | 核心产出是 9:20 晨报，而非实时推送流 |
| **N_FEATURES 合同** | 每新增特征组，同步更新 C++ `FeatureExtractor` 和 Python `feature_builder.py` |

## 五、Phase 优先级 & 依赖关系

```
Phase 7（晨报 + 窗口评分）← 基础，其他 Phase 都依赖它
        ↓
Phase 8（博弈信号）← 增强晨报标的评分
Phase 9（宏观跨资产）← 增强晨报宏观环境板块
        ↓
Phase 10（对抗决策）← 依赖 Phase 7/8/9 的信号作为输入
        ↓
Phase 11（另类数据）← 独立，可并行或延后
```

**建议执行顺序：**
1. **Phase 7**（必须先做，是一切的基础）
2. **Phase 8 + Phase 9**（可并行）
3. **Phase 10**（依赖前三个 Phase 的信号质量）
4. **Phase 11**（锦上添花，可延后）

## 六、成功标准

| 指标 | 目标 |
|------|------|
| 晨报生成时间 | 每个交易日 9:20 前稳定产出 |
| 晨报阅读时间 | ≤5 分钟决策完毕 |
| "不做"信号占比 | ≥70%（大多数时候系统应该闭嘴） |
| 判决书可解释性 | 每个信号必须包含概率+根因+反驳+行动条件 |
| 预测回溯覆盖率 | 100% 的信号在 T+20 后自动回溯 |
| 测试覆盖 | 新增 C++ 模块均有对应 unit test |
